# Derrops Conventions

Every AWS resource name, CloudFormation export, S3 bucket path, IAM policy ARN, and CDK tag in this platform is generated by the `@derrops-conventions` TypeScript library via the `DerropsConventions` class. This page describes how conventions are applied across the platform — the domain model, the tag schema, and the CDK patterns.

For the rationale behind the naming and segregation strategy, see the [Derrops Guide to Naming Conventions and Segregation](/blog/derrops-conventions).

---

## Org-level setup

All convention instances derive from a single org-level root, initialized once per CDK app in `packages/derrops-config/src/derrops-conventions.ts`:

```typescript
const orgConvention = new DerropsConventions({ org: 'derrops', env, region })
  .arnContext({ accountId })
  .tagPrefix('derrops:')
  .tagKeys('org', 'domain', 'service')
```

Domain- and service-level instances are derived from this root via `.with()`, which is immutable — each call returns a new instance:

```typescript
const platform = orgConvention.with({ domain: 'platform' })
const vpc = platform.with({ service: 'vpc' })
```

See [Domain & Service Registry](./service-registry) for per-domain instantiation patterns and every `domain + service` combination.

## Domains

A **domain** is a named functional boundary that groups related AWS resources, services, and documentation under a single area of responsibility. It is the second segment in every resource name and the value of the `derrops:domain` tag.

| Domain            | CDK tag value (`derrops:domain`) | Responsibility                                                                         |
| ----------------- | -------------------------------- | -------------------------------------------------------------------------------------- |
| `platform`        | `platform`                       | Shared networking, compute, and storage that all other domains build on                |
| `user-management` | `user-management`                | Who can access the platform — sign-up, sign-in, session tokens, and machine identity   |
| `security`        | `security`                       | Cross-cutting controls — encryption, certificates, firewall rules, and compliance      |
| `oaspec`          | `oaspec`                         | Storage, indexing, and retrieval of OpenAPI specification documents                    |
| `relay`           | `relay`                          | Traffic capture and forwarding — proxy components that route API calls to the platform |
| `obs`             | `obs`                            | The pipeline that turns raw HTTP captures into enriched, queryable log records         |
| `portal`          | `portal`                         | The customer-facing web experience — dashboards, configuration, analytics, and alerts  |

### Domain responsibilities

**`platform`** — Foundational resources every other domain builds on. No feature domain takes a hard dependency on another feature domain's internals; those shared primitives go here instead. Includes: VPC, Aurora cluster, OpenSearch collection, API Gateway, security groups, Route 53 private hosted zone, core API Lambda.

**`user-management`** — Resources that decide _who_ can access the platform or _what identity_ an automated client presents. Includes: Cognito User Pool and App Client, Cognito Identity Pool for relay IAM identity, pre-token-generation Lambda (custom claims), user profile and tenant membership store.

**`security`** — Cross-cutting controls whose primary job is protection rather than feature delivery. Referenced by other domains; owned here. Includes: KMS keys, ACM certificates, Secrets Manager entries, WAF WebACL, AWS Config and Security Hub.

**`oaspec`** — Full OASpec lifecycle: ingest → validate → index → search. Includes: Derrops-managed source bucket, per-tenant spec storage, upload staging, OpenSearch indexer Lambda, DynamoDB hot-path cache.

**`relay`** — Runs inside or alongside the customer's network. Includes: Cloud Relay proxy, local development relay, Aegis Token Broker, per-relay IAM identity, relay instance registry.

**`obs`** — Handles HTTP log data after it leaves a relay. Includes: log ingestion endpoint, OASpec enrichment pipeline, per-tenant S3 log buckets, Aurora tables, log query and aggregation services.

**`portal`** — Customer-facing web experience. Includes: React web app, dashboards, SLA trend charts, API performance metrics, cost analysis, alert management, relay configuration UI, OASpec catalogue browser.

## Services

A **service** is the second-level unit within a domain — it maps to a single deployable unit or coherent group of AWS resources and becomes the value of the `derrops:service` tag. Every resource name is `derrops--{domain}--{service}[--{key}]`.

See [Domain & Service Registry](./service-registry) for generated resource name patterns and per-domain `DerropsConventions` instantiation examples.

### Adding a new service

1. Confirm no existing service already covers the use case.
2. Choose a kebab-case name that describes the capability, not the implementation (`storage`, not `s3-storage`).
3. Add it to the table above and to [Domain & Service Registry](./service-registry).
4. Instantiate via `domainConvention.with({ service: 'my-service' })` and apply tags with `applyTags()`.
5. If the name is referenced across packages, define it in `packages/derrops-config/src/`.

## Adding a new domain

Adding a domain affects tagging, documentation, billing, and enforcement. Before creating one, verify the new area cannot sit inside an existing domain. If it genuinely cannot:

1. Add it to the domain table above and to [Domain & Service Registry](./service-registry).
2. Add the domain tag to `apps/derrops-docs/internal/platform/design/tags.yml`.
3. Update `tagKeys` on the org-level convention instance in `packages/derrops-config/src/derrops-conventions.ts`.
4. Update the domain tag reference in `apps/derrops-docs/internal/platform/design/CLAUDE.md`.
5. Deploy a CDK Config rule update so the new value is recognised as valid.

## Tags

All tags are generated by `DerropsConventions` — never set manually. Apply them in CDK via:

```typescript
svcConvention.applyTags((k, v) => Tags.of(this).add(k, v))
```

The `tagPrefix` (`derrops:`) and `tagKeys` (`org`, `domain`, `service`) configured on the org-level instance propagate through every `.with()` derivative.

### Required tags

Every Derrops resource — shared platform infrastructure and per-tenant resources alike — must carry these five tags:

| Tag key              | Value                        | Purpose                                                                                  |
| -------------------- | ---------------------------- | ---------------------------------------------------------------------------------------- |
| `derrops:org`        | `derrops`                    | Top-level ownership boundary; enables org-wide queries in Security Hub and Cost Explorer |
| `derrops:env`        | `prod` \| `dev` \| `staging` | Deployment stage; used in billing dashboards and runbook targeting                       |
| `derrops:domain`     | Kebab-case domain name       | Business capability boundary; aligns with the `{domain}` segment in naming conventions   |
| `derrops:service`    | Kebab-case service name      | Deployable unit identity; enables per-service cost and security grouping                 |
| `derrops:managed-by` | `cdk`                        | Signals the resource is IaC-managed; manual changes are a drift signal                   |

### Tenant tag

Resources that belong to a specific tenant carry one additional tag:

| Tag key             | Value format     | Purpose                                                                       |
| ------------------- | ---------------- | ----------------------------------------------------------------------------- |
| `derrops:tenant-id` | Opaque tenant ID | Tenant attribution for cost allocation, IAM condition keys, and audit queries |

Shared platform resources (VPC, Aurora cluster, API Gateway, OpenSearch collection) do **not** carry `derrops:tenant-id`. The reserved value `derrops` is used for platform-owned resources such as the managed OASpec index.

:::note
`derrops:tenant-id` uses the opaque ID, never a human-readable name. See [Multi-Tenancy — Tenant ID format](./multi-tenancy#tenant-id-format) for why.
:::

### Optional tags

| Tag key                       | Value format                             | When to add                                                                                 |
| ----------------------------- | ---------------------------------------- | ------------------------------------------------------------------------------------------- |
| `derrops:component`           | Kebab-case label                         | When a service has multiple independently-operable sub-components (e.g. `opensearch-index`) |
| `derrops:cost-centre`         | Cost centre code                         | When finance requires per-tenant or per-domain billing attribution                          |
| `derrops:data-classification` | `public` \| `internal` \| `confidential` | On storage resources (S3, RDS, DynamoDB, OpenSearch) that hold customer data                |

### Tag examples

**Shared platform resource (Aurora cluster)**

```
derrops:org          = derrops
derrops:env          = prod
derrops:domain       = platform
derrops:service      = app-database
derrops:managed-by   = cdk
```

**Per-tenant OASpec S3 bucket** (name: `us-east-1--prod--derrops--t-a3f8b2--oaspec--storage--specs`)

```
derrops:org          = derrops
derrops:env          = prod
derrops:domain       = oaspec
derrops:service      = storage
derrops:managed-by   = cdk
derrops:tenant-id    = t-a3f8b2
```

**Platform-managed OASpec index** (reserved `derrops` tenant)

```
derrops:org          = derrops
derrops:env          = prod
derrops:domain       = oaspec
derrops:service      = indexer
derrops:managed-by   = cdk
derrops:tenant-id    = derrops
```

## CDK enforcement

### Tag propagation caveat

Apply `applyTags()` at the highest scope that makes sense. Stack-level tags cover all child resources automatically — this is preferred to guarantee no resource is missed. Use construct-level tags only when a sub-tree needs a different value (e.g. `derrops:tenant-id`).

Some AWS resources do not inherit tags from their parent CloudFormation stack:

- **OpenSearch Serverless collections** — set tags directly on the `CfnCollection` resource.
- **S3 object prefixes** — objects inherit bucket tags only if explicitly configured; set object-level tags via the upload handler for per-object attribution.
- **IAM resources** — inherit stack tags normally.

Verify propagation after deployment with `aws resourcegroupstaggingapi get-resources`.

### CDK Aspect

`RequiredTagsAspect` in `packages/derrops-infra/lib/aspects/` validates required tags at synth time. A missing tag throws `Annotations.of(node).addError()`, catching omissions before deployment:

```typescript
// bin/cdk.ts
Aspects.of(app).add(
  new RequiredTagsAspect({
    required: [
      'derrops:org',
      'derrops:env',
      'derrops:domain',
      'derrops:service',
      'derrops:managed-by',
    ],
  }),
)
```

### AWS Config rule

`derrops--platform--compliance--required-tags-rule` flags any resource missing the five required tags. Violations appear in Security Hub and trigger a Slack alert to the platform team.

## Cost allocation

AWS Cost Explorer splits costs by:

1. `derrops:env` — environment-level billing (prod vs dev)
2. `derrops:domain` — domain-level spend tracking
3. `derrops:tenant-id` — per-tenant cost attribution (enables per-tenant billing)

All five required tags are activated as Cost Allocation Tags in the AWS billing console. New tags must be activated there before they appear in Cost Explorer reports.

## IAM condition keys

Tags enable IAM conditions that enforce tenant isolation without per-tenant policy documents:

```json
{
  "Condition": {
    "StringEquals": {
      "aws:ResourceTag/derrops:tenant-id": "${aws:PrincipalTag/derrops:tenant-id}"
    }
  }
}
```

Security Hub and GuardDuty findings are filtered by `derrops:service` and `derrops:tenant-id` to scope incidents to the affected service and tenant.

## Domains in documentation

Every design doc carries the matching domain tag in its frontmatter alongside any topic tags. This lets readers navigate all content for a domain at `/internal/platform/design/tags/{domain}`:

```yaml
tags:
  - component-design
  - oaspec
```

Doc tags and AWS resource tags (`derrops:domain`, `derrops:service`, etc.) are different systems. The domain names overlap by convention, but doc tags are plain lowercase strings in YAML frontmatter — never use the `derrops:key` colon-separated format in a doc file.
